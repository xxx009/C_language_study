# 基于图结构的城市地铁换乘规划与运营优化系统

## 【功能描述】

设计并实现一个城市地铁换乘规划与运营优化系统。系统以地铁站点、线路、线路站点顺序、站间运行时间、换乘关系等数据为基础，使用图结构对城市地铁网络进行建模，并通过路径规划算法为用户提供从起点站到终点站的推荐出行路线、预计耗时、换乘次数、收益估算和运营成本估算。

系统应支持地铁基础数据维护，包括站点维护、线路维护、线路站点顺序维护、换乘站点维护、列车运行时间维护、JSON 数据导入导出等功能。线路图不能只依赖 `stations` 和 `lines` 两类数据，还需要通过 `line_stations` 描述每条线路上站点的先后顺序，从而支持普通线路、环线、支线、分段运营和暂停运营站点等真实场景。

系统还应预留线路选择优化接口，可根据不同策略选择最短时间、最少换乘、最低成本或综合最优路线。运营分析部分应支持乘客数量输入，并按照“每相邻 1 站每人收入 0.1 元、每相邻 1 站每人成本 0.4 元”的规则估算本次行程收益和运营成本。

## 【具体要求】

### 1、数据结构设计

#### ① 图结构存储地铁网络（必做）

使用图结构表示地铁网络：

- 顶点：地铁站点。
- 边：相邻站点之间的运行区间或换乘通道。
- 边权：站间运行时间、换乘时间或综合代价。
- 路径结果：保存经过站点、总耗时、换乘次数和是否可达。

建议设计以下结构体：

- `station`：存储站点编号、站点名称、是否启用。
- `line`：存储线路编号、线路名称、站点顺序和站点数量。
- `transfer`：存储换乘点编号、换乘两端站点、换乘时间、是否启用。
- `edge`：存储边的目标站点、耗时、所属线路、是否为换乘边。
- `route_result`：存储路径规划结果。
- `subway_graph`：作为全局地铁网络结构，统一管理站点、线路、换乘点和边。

#### ② 线路站点顺序结构（必做）

仅有 `stations` 和 `lines` 不足以表达线路图，因为无法体现某条线路上的站点先后顺序。因此 JSON 文件中必须增加 `line_stations` 字段。

推荐格式如下：

```json
{
  "line_stations": [
    {
      "line_id": 0,
      "line_name": "1号线",
      "segment_id": 0,
      "segment_name": "主线",
      "active": 1,
      "station_ids": [0, 1, 2, 3, 4],
      "station_names": ["西塱", "坑口", "花地湾", "芳村", "黄沙"]
    }
  ]
}
```

`line_stations` 应支持：

- 普通线路；
- 分支线路；
- 分段运营线路；
- 环线线路；
- 同一线路多个运营区段；
- 暂停运营站点；
- 线路站点名称校验。

例如，环线可以让 `station_ids` 首尾相同；分段运营可以使用同一个 `line_id` 和不同 `segment_id` 表示多个区段。

#### ③ 换乘点独立维护（必做）

换乘点应作为独立数据对象维护，而不是简单混入普通线路区间。每条换乘记录至少包含：

- 换乘编号；
- 起始站点；
- 目标站点；
- 换乘耗时；
- 是否启用。

示例：

```json
{
  "transfers": [
    {"a": 13, "b": 50, "minutes": 5}
  ]
}
```

#### ④ 运营与成本扩展结构（选做）

为了支持后续运营分析，可扩展以下结构：

- 每条线路的列车数量；
- 每条线路不同时间段的发车间隔；
- 每个站点的客流分布；
- 固定运营成本；
- 按公里或按站点计算的变动成本；
- 预算上限；
- 收益与成本报告。

客流分布可支持：

- 均匀分布；
- 泊松分布；
- t 分布；
- 正态分布；
- 基于历史数据的预测模型。

### 2、功能点要求

#### ① 地铁站点维护（必做）

系统应支持：

- 新增站点；
- 删除站点；
- 修改站点名称；
- 设置站点启用或停用；
- 查询站点编号和站点名称；
- 暂停运营站点不参与路径规划。

#### ② 地铁线路维护（必做）

系统应支持：

- 新增线路；
- 删除线路；
- 修改线路名称；
- 查询线路信息；
- 按线路输出站点列表。

#### ③ 线路站点顺序维护（必做）

系统应支持：

- 根据 `line_stations` 配置线路经过站点；
- 支持主线、支线、分段线路；
- 支持环线；
- 支持线路区段启用或停用；
- 根据站点顺序自动生成相邻站点运行边。

#### ④ 站间运行时间维护（必做）

系统应支持维护相邻站点之间的运行时间。

数据可通过 `segments` 字段表示：

```json
{
  "segments": [
    {"a": 0, "b": 1, "line": "1号线", "minutes": 3},
    {"a": 1, "b": 2, "line": "1号线", "minutes": 4}
  ]
}
```

如果未提供运行时间，可默认相邻站点为固定时间，例如 3 分钟。

#### ⑤ 换乘关系维护（必做）

系统应支持：

- 新增换乘关系；
- 删除换乘关系；
- 修改换乘耗时；
- 查询某站是否为换乘站；
- 路径规划时将换乘时间计入总耗时。

#### ⑥ JSON 文件导入与导出（必做）

系统应支持通过 JSON 文件导入和导出地铁网络数据。

JSON 文件建议包含：

- `metadata`：城市、系统名称、更新时间、说明；
- `stations`：站点基础数据；
- `lines`：线路基础数据；
- `line_stations`：线路站点顺序；
- `segments`：相邻站运行时间；
- `transfers`：换乘关系；
- `fleet_profiles`：线路配车信息（选做）；
- `operation_windows`：运营时间与发车间隔（选做）。

系统应能兼容基础 JSON 和扩展 JSON。若出现两个 JSON 对象拼接的情况，程序应能给出错误提示或只读取第一个合法对象。

#### ⑦ 地图展示功能（必做）

系统应将地铁线路按行列输出，尽量美观清晰。

要求：

- 按线路输出；
- 每条线路按站点顺序输出；
- 每行可配置输出列数；
- 换乘站可用特殊符号标记；
- 停运站点可显示为“暂停”。

#### ⑧ 最优路线规划（必做）

用户输入起点站和终点站后，系统应输出：

- 是否可达；
- 推荐路线；
- 经过站点；
- 总耗时；
- 换乘次数；
- 本次行程收益；
- 本次行程运营成本。

路线规划算法要求：

- 使用 Dijkstra、Floyd、A* 或 BFS 等算法；
- 推荐使用 Dijkstra 算法处理带权图；
- 边权至少包含站间运行时间；
- 换乘边应加入换乘时间或换乘惩罚；
- 支持自定义代价函数，便于扩展最少换乘、最低成本等策略。

#### ⑨ 收益与成本计算（必做）

系统应根据路径经过的相邻站点数量和乘客人数计算收益与成本。

规则如下：

- 每相邻 1 站，每名乘客收入增加 0.1 元；
- 每相邻 1 站，每名乘客运营成本增加 0.4 元；
- 总收益 = 相邻站点数 × 0.1 × 乘客数；
- 总成本 = 相邻站点数 × 0.4 × 乘客数。

示例：

若路径经过 5 个相邻站点，乘客数为 20，则：

- 收益 = 5 × 0.1 × 20 = 10 元；
- 成本 = 5 × 0.4 × 20 = 40 元。

### 3、融合 AI 的任务要求

#### ① 自然语言路线查询（选做）

系统可支持用户使用自然语言输入出行需求，例如：

- “我要从体育西路到广州南站，尽量少换乘。”
- “从西塱到珠江新城最快怎么走？”
- “避开机场南站，帮我规划路线。”
- “预算优先，选择成本最低的路线。”

AI 模块应将自然语言转换为结构化查询：

```json
{
  "start": "体育西路",
  "end": "广州南站",
  "preference": "min_transfer"
}
```

#### ② AI 路线推荐解释（选做）

在输出路线时，AI 可以给出解释性文字，例如：

> 推荐该路线是因为总耗时较短，换乘次数较少，并且避开了暂停运营站点。

#### ③ AI 数据纠错（选做）

AI 可辅助检查 JSON 数据问题：

- 线路引用了不存在的站点 ID；
- `station_ids` 与 `station_names` 数量不一致；
- 同一个站点重复定义；
- 环线首尾站点不一致；
- 分支线路缺少 `segment_id`；
- 停运站点仍被规划为可乘坐节点。

#### ④ AI 客流预测（选做）

AI 可根据历史客流、时段、节假日等因素预测不同站点的乘客数量。

预测结果可用于：

- 调整发车间隔；
- 增减列车数量；
- 预警拥挤站点；
- 计算收益和成本。

#### ⑤ AI 运营优化建议（选做）

AI 可根据预算、客流和线路运行情况给出建议：

- 哪条线路应增加列车；
- 哪些时段应缩短发车间隔；
- 哪些低客流时段可降低运力；
- 如何在预算内提升服务质量；
- 如何减少运营成本。

#### ④ 大规模真实数据扩展（选做）

可导入广州地铁或其他城市地铁数据，支持：

- 多城市；
- 多线路；
